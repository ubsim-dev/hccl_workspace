结论

  • 随着不均衡从 1.2x 增加到 3.0x，baseline 单 rank 带宽从 227.47 降到 212.07 GB/s，下降约 6.8%。
  • Snack v1 基本稳定在 89 GB/s 左右，没有随更强不均衡获得收益；v0 更低，约 79-88 GB/s。
  • v1 的 plane CV 很低，说明平面字节量分配不是主要问题；主要成本来自 source-plane 和 dest-plane 的静态依赖串行化。
  • 这个模型保留 directed plane 语义，因此没有用删除 TP 的 strict 方式；如果要对齐老 MeshClos strict，需要把正反方向 plane 约束成 pair-symmetric。

原因分析

  • baseline 不是单平面模型。当前 baseline 的 transport_channel.csv 对跨组 peer 保留 4 条 TP，同一个大 peer 流可以被 ns3-ub 的 full-TP 口径分摊到多条路径上；因此即使单条流 max/min 从 1.2x 增大到 3.0x，baseline 仍能维持 212-227 GB/s/rank。
  • snack v0/v1 是 directed fixed-plane。这里用 actualPlane 映射为 priority 3/4/5/6，让每个跨组 task 只命中一个平面。这可以表达脚本里的有向平面选择，但每条 peer 流天然被限制在单 TP 上，单流上限接近单平面能力。
  • 主要瓶颈不是平面总量不均。v1 的 plane CV 在 1.2x/1.5x/2.0x/3.0x 下分别只有 0.001/0.004/0.006/0.009，说明四个平面的总字节量已经很均衡；但是带宽仍卡在约 89 GB/s/rank。
  • 真正的代价来自静态依赖串行化。snack 对同一个 source + actualPlane 串发送队列，同时对同一个 dest + actualPlane 串接收互斥队列。表里的 Max dst-plane queue 到 7-8，意味着某些目的 rank 的同一平面上有 7-8 个接收 task 被串起来，makespan 被最长依赖链拉长。
  • 不均衡越强，baseline 只轻微下降，snack 的 makespan 线性变长。从 1.2x 到 3.0x，baseline makespan 75.48us -> 148.82us；snack-v1 是 188.99us -> 353.35us。v1 的单 rank 带宽几乎不变，说明它已经被“单平面 + 依赖链”限制住，而不是被随机流量分布限制住。
  • 和老 MeshClos strict 口径不同。老 strict 是在 transport_channel.csv 里对每个无向 pair 直接删到只剩 1 条 TP；而 snack 脚本里 src->dst 和 dst->src 经常选不同 plane，所以不能只靠删 TP 表达。若要严格复用老 strict 口径，需要把平面选择改成 pair-symmetric。
